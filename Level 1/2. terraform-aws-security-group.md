# Terraform: AWS Security Group (datacenter-sg) on LocalStack

A Terraform configuration that provisions an AWS EC2 security group under the default VPC (against a LocalStack backend) with inbound rules for HTTP (port 80) and SSH (port 22) from anywhere. Part of the Nautilus DevOps team's incremental AWS migration.

---

## Task

The Nautilus DevOps team is strategizing the migration of a portion of their infrastructure to the AWS cloud. Recognizing the scale of this undertaking, they have opted to approach the migration in incremental steps rather than as a single massive transition. To achieve this, they have segmented large tasks into smaller, more manageable units. This granular approach enables the team to execute the migration in gradual phases, ensuring smoother implementation and minimizing disruption to ongoing operations. By breaking down the migration into smaller tasks, the Nautilus DevOps team can systematically progress through each stage, allowing for better control, risk mitigation, and optimization of resources throughout the migration process.

Use Terraform to create a security group under the default VPC with the following requirements:

1. The name of the security group must be `datacenter-sg`.
2. The description must be `Security group for Nautilus App Servers`.
3. Add an inbound rule of type `HTTP`, with a port range of `80`, and source CIDR range `0.0.0.0/0`.
4. Add another inbound rule of type `SSH`, with a port range of `22`, and source CIDR range `0.0.0.0/0`.

Ensure that the security group is created in the `us-east-1` region using Terraform. The Terraform working directory is `/home/bob/terraform`. Create the `main.tf` file (do not create a different `.tf` file) to accomplish this task.

**Note:** Right-click under the `EXPLORER` section in `VS Code` and select `Open in Integrated Terminal` to launch the terminal.

## Key Design Choices

The configuration uses three resources to satisfy the requirements:

1. **`data "aws_vpc" "default"`** looks up the default VPC ID at apply time rather than hardcoding it. LocalStack provisions a default VPC automatically, but its ID is not stable across labs.
2. **`aws_security_group`** creates the security group itself with the required name, description, and default VPC association. The inbound rules are deliberately **not** defined inline here.
3. **Two separate `aws_security_group_rule`** resources, one for HTTP and one for SSH, attach the inbound rules to the security group.

The separate-rules pattern is preferred over inline `ingress {}` blocks for two reasons. First, the task lists the two rules as discrete requirements, and modeling them as discrete resources gives each one its own state entry and lifecycle. Second, mixing inline `ingress` blocks with `aws_security_group_rule` resources on the same SG is a well-known cause of Terraform drift, where Terraform repeatedly tries to add and remove the same rule on every apply. Keeping all rules as standalone resources avoids the trap entirely.

## Solution

`/home/bob/terraform/main.tf`:

```hcl
# Look up the default VPC in the configured region
data "aws_vpc" "default" {
  default = true
}

# Create the security group in the default VPC
resource "aws_security_group" "datacenter_sg" {
  name        = "datacenter-sg"
  description = "Security group for Nautilus App Servers"
  vpc_id      = data.aws_vpc.default.id
}

# Inbound rule: HTTP (port 80) from anywhere
resource "aws_security_group_rule" "http_inbound" {
  type              = "ingress"
  from_port         = 80
  to_port           = 80
  protocol          = "tcp"
  cidr_blocks       = ["0.0.0.0/0"]
  security_group_id = aws_security_group.datacenter_sg.id
  description       = "HTTP from anywhere"
}

# Inbound rule: SSH (port 22) from anywhere
resource "aws_security_group_rule" "ssh_inbound" {
  type              = "ingress"
  from_port         = 22
  to_port           = 22
  protocol          = "tcp"
  cidr_blocks       = ["0.0.0.0/0"]
  security_group_id = aws_security_group.datacenter_sg.id
  description       = "SSH from anywhere"
}
```

Notice what is **not** in this file:

- No `terraform { required_providers }` block. The AWS provider is already pinned in `provider.tf`.
- No `provider "aws"` block. Already configured in `provider.tf`.
- No `region` argument. Inherited from the provider configuration.

## Explanation of the Solution

### 1. Default VPC Lookup

```hcl
data "aws_vpc" "default" {
  default = true
}
```

The task says to create the security group "under the default VPC." Rather than hardcoding a VPC ID, this data source asks the provider for the VPC that has the `IsDefault` flag set in the target region. LocalStack provisions a default VPC automatically at startup, and this lookup returns its ID (`vpc-fe0de1421fbdb50a2` in this run). Using a data source keeps the configuration portable: the same `main.tf` works against real AWS, LocalStack, or any other AWS-compatible backend without changes.

### 2. The Security Group Resource

```hcl
resource "aws_security_group" "datacenter_sg" {
  name        = "datacenter-sg"
  description = "Security group for Nautilus App Servers"
  vpc_id      = data.aws_vpc.default.id
}
```

The `name` and `description` arguments are set to the exact strings the task requires. Both are immutable: changing either after creation forces Terraform to destroy and recreate the security group. The `vpc_id` is interpolated from the data source output.

Note that the description is also required by AWS itself, not just the task. AWS rejects security groups created without a description. The provider validates this client-side too.

### 3. Inbound Rules as Separate Resources

```hcl
resource "aws_security_group_rule" "http_inbound" {
  type              = "ingress"
  from_port         = 80
  to_port           = 80
  protocol          = "tcp"
  cidr_blocks       = ["0.0.0.0/0"]
  security_group_id = aws_security_group.datacenter_sg.id
}
```

Each rule is its own resource rather than an inline `ingress {}` block inside `aws_security_group`. This pattern has two advantages:

- **Avoids drift.** If you ever mix inline `ingress` blocks with `aws_security_group_rule` resources on the same SG, Terraform sees them as competing definitions and will repeatedly add and remove the same rule on each apply. Using only `aws_security_group_rule` resources eliminates that risk.
- **Cleaner state and easier changes.** Each rule has its own state entry, can be modified or destroyed independently, and shows up as a discrete plan line.

The protocol is `tcp` because both HTTP and SSH run over TCP. `from_port` and `to_port` are both set to the same value (80 or 22) because each rule covers a single port rather than a range. The CIDR `0.0.0.0/0` matches all IPv4 addresses, satisfying the "from anywhere" requirement.

### Why no egress rule?

The task only specifies inbound rules. The `aws_security_group` resource creates a default egress rule allowing all outbound traffic unless you explicitly override it. That default is the standard AWS behavior, and matches the task's silence on egress.

## Common Pitfalls

| Symptom | Likely Cause | Fix |
|---|---|---|
| `Error: Duplicate provider configuration` during `init` or `plan` | `main.tf` also declares `provider "aws"`, conflicting with `provider.tf` | Remove the `provider "aws"` block from `main.tf`. The pre-existing `provider.tf` is authoritative. |
| `Error: no matching VPC found` from the data source | The default VPC was deleted, or LocalStack started without one | Restart LocalStack, or fall back to creating a VPC explicitly with `aws_vpc` and referencing it |
| Apply succeeds but rules are missing on `describe-security-groups` | Mixed inline `ingress {}` blocks with `aws_security_group_rule` resources, causing them to overwrite each other | Use one pattern only. The solution above uses standalone `aws_security_group_rule` resources exclusively. |
| `InvalidParameterValue: Security group description must not be empty` | Forgot the `description` argument | Set `description = "Security group for Nautilus App Servers"` exactly as the task specifies |
| `InvalidGroup.Duplicate: The security group 'datacenter-sg' already exists` | A previous apply left the SG in LocalStack | `terraform destroy` then re-apply, or `terraform import aws_security_group.datacenter_sg sg-xxxxxxxx` |
| Rules apply in the wrong direction (egress instead of ingress) | `type` set to `egress` by mistake | Set `type = "ingress"` on both rule resources |
| `protocol` set to `"HTTP"` or `"SSH"` causes a validation error | These are not valid IP protocol values | Use `protocol = "tcp"`. HTTP and SSH are application-layer protocols that ride on TCP; AWS rules operate at the transport layer |
| The grader fails despite everything looking correct | The grader checks state at validation time; do not run `terraform destroy` before clicking validate | Re-run `terraform apply` and click validate before destroying |
